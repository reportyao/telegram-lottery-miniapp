# 塔吉克斯坦索莫尼(TJS)显示格式与移动应用本地化实践蓝图

## 1. 概述与结论先行(TL;DR)
塔吉克斯坦的法定货币为索莫尼(Somoni),国际标准化代码为TJS,数字代码972,辅币单位为迪拉姆(Diram),进制为1索莫尼=100迪拉姆。这意味着在通用财务显示中,索莫尼应采用两位小数,以体现到“分”(迪拉姆)级别的精确性[^1][^8][^4]。

本地化数字格式方面,塔吉克语(Tajik)习惯采用阿拉伯数字,小数点使用逗号“,”,千位分隔符使用不间断空格“ ”,并按三位一组进行分组。特别地,整数部分为4位数时可以省略千位分隔符(如“25000”可显示为“25 000”或“25000”,两者均可)[^5][^10]。

在界面显示中,货币符号/代码的使用需兼顾可读性与无歧义。国际平台示例显示使用符号“SM”或ISO代码“TJS”的都有,且存在不同的本地化格式示例(如Salesforce Help中“1 234 567,57 сом”)[^7][^3]。在移动端小屏环境下,建议默认使用ISO代码TJS后缀(与金额间保留一个不间断空格),在用户已熟悉符号或空间受限的场景可切换至SM,并确保负数、零值、空值、一屏多货币等边界情况表现一致[^6][^2]。

## 2. 研究方法与来源质量说明
本研究围绕“官方与事实标准”“平台实现”“本地化实践”三类来源展开,综合取用以达到工程可落地的结论。

首先,事实与背景采用百科类与标准页面:塔吉克斯坦索莫尼的货币属性(代码、单位、引入历史等)来自Wikipedia条目,辅以ISO 4217官方页面确证货币代码与数字代码的标准属性[^1][^8]。其次,平台实现参考Salesforce的帮助页面(列出支持货币及ICU样例),以及微软全球化与ICU文档,以了解在通用平台中如何落地本地化格式化[^7][^11][^9]。第三,塔吉克语数字与货币格式的本地化规范依据Microsoft Tajik Style Guide,这是面向本地化实践的权威风格指南[^5]。

## 3. 基础定义:货币与单位
塔吉克斯坦索莫尼(Somoni,ISO 4217代码TJS;数字代码972)于2000年10月30日发行,用以取代塔吉克卢布,辅币单位为迪拉姆(Diram),进制关系为1索莫尼=100迪拉姆[^1]。在货币精度层面,这意味着应使用两位小数进行财务计算与显示,以准确表达至“分”的层级[^4]。

表2:TJS基础属性
| 属性 | 值 |
|---|---|
| ISO三字母代码 | TJS[^1][^8] |
| ISO数字代码 | 972[^8] |
| 符号(常见) | SM(常见于国际平台示例)[^7][^3] |
| 货币名称(中文/英文) | 索莫尼 / Somoni[^1] |
| 辅币单位 | 迪拉姆(Diram)[^1] |
| 进制关系 | 1索莫尼=100迪拉姆[^1] |
| 推荐显示小数位数 | 2位(基于辅币进制)[^4] |

上述属性构成后续数字与货币格式化的底层约束,尤其是小数位采用两位的设定,直接决定了界面显示与数值计算的统一。

## 4. 本地化数字与货币显示规范(塔吉克语环境)
塔吉克语数字使用阿拉伯数字(0–9),小数点为逗号“,”,千位分隔符为不间断空格“ ”,分组规则为三位一组。风格指南明确:整数部分为4位数时,可省略千位分隔符;非正规文本或工具生成值(如Excel导出)可不强制带千位分隔符[^5]。此外,小数分隔符与列表分隔符在风格指南中分别示例为逗号与分号(如“25 000,45”的小数用逗号;列表示例使用分号),这有助于在工程实现中明确解析与显示的边界[^5]。

在货币显示中,符号位置与空格处理对可读性与准确性至关重要。不同来源对TJS显示样例存在差异:国际平台(Salesforce Help)给出了“1 234 567,57 сом”(使用“сом”词形、逗号与不间断空格、两位小数);XE等常见货币工具采用“TJS 代码 + SM 符号”说明,强调代码与符号在国际场景的通用识别力[^7][^3]。在工程实践中,建议默认采用“金额 + 不间断空格 + TJS”的方式,既明确币种,又减少跨区域歧义;在用户熟悉符号、空间受限或特定本地化场景,可回退到“金额 + 不间断空格 + SM”。

表3:塔吉克语数字与货币格式化规则(摘要)
| 规则项 | 塔吉克语(tg_TJ)建议 |
|---|---|
| 小数点 | 逗号“, ”[^5] |
| 千位分隔符 | 不间断空格“ ”(三位一组)[^5] |
| 4位整数 | 可省略千位分隔符(如“25000”可显示为“25 000”或“25000”)[^5] |
| 列表分隔符 | 分号“;”(示例)[^5] |
| 货币符号位置 | 建议后缀(金额后空一格),与代码配合使用提高辨识度[^5][^7] |
| 小数位数 | 2位(基于辅币迪拉姆)[^1][^4] |

表4:TJS显示示例矩阵(建议)
| 场景 | 格式 | 示例 |
|---|---|---|
| 一般显示(正数) | 金额 + 不间断空格 + TJS | 1 234 567,57 TJS |
| 一般显示(负数) | -金额 + 不间断空格 + TJS | -1 234 567,57 TJS |
| 简洁显示(正数) | 金额 + 不间断空格 + SM | 1 234 567,57 SM |
| 简洁显示(负数) | -金额 + 不间断空格 + SM | -1 234 567,57 SM |
| 紧凑显示(正数) | 金额 + 空格 + TJS | 1 234 567,57 TJS |
| 紧凑显示(负数) | -金额 + 空格 + TJS | -1 234 567,57 TJS |
| 4位整数(无千分位) | 金额(无分隔符) + 空格 + TJS | 25 000,00 TJS 或 25000,00 TJS[^5] |

表5:ICU/Salesforce示例对照(解释性)
| 来源 | 示例 | 观察 |
|---|---|---|
| Salesforce Help(tg_TJ)[^7] | 1 234 567,57 сом | 使用“сом”本地词形,逗号作小数,不间断空格三位分组 |
| 其他平台(XE等)[^3] | TJS(代码)+ SM(符号) | 国际场景强调ISO代码与符号并存以增强辨识 |

上述矩阵强调:在工程实现中,应通过区域敏感格式化(ICU/CLDR)确保逗号与不间断空格的正确使用,同时保留用户覆盖选项,以应对不同业务场景对简洁性与可读性的取舍[^9][^11][^6]。

## 5. 移动端(iOS/Android)最佳实践
在移动端场景,核心目标是“与用户语言/区域一致”的格式化与“无歧义”的币种标识。实现路径包括:系统区域检测与用户显式选择相结合;使用平台的原生NumberFormatter/NSNumberFormatter与ICU/CLDR模式进行金额格式化;采用货币代码而非仅凭符号来消解歧义;针对空间受限界面设计符号/代码切换策略;系统性覆盖负数、零值、空值与多货币并列等边界情况[^6][^11][^9]。

表6:平台格式化API选择与用法(概览)
| 平台 | 核心API | 关键用法 |
|---|---|---|
| Android | NumberFormatter(Jetpack)[^11][^9] | setCurrency(TJS)/setPattern/不间断空格与分组;locale = "tg_TJ"或跟随系统 |
| iOS | NSNumberFormatter[^11][^9] | currencyCode = "TJS";currencySymbol = "SM"可覆盖;不间断空格与分组随locale |
| 跨端 | ICU/CLDR模式[^9] | 通过locale模式与自定义模式确保逗号、小数位与分组一致 |

在货币标识策略上,建议默认采用“金额 + 不间断空格 + TJS”,在用户偏好中提供“仅SM/仅TJS/自动”三种选项,以兼顾国际可读性与本地简洁性。

表7:移动端UI模式与适配策略
| 模式 | 优点 | 风险 | 适用场景 |
|---|---|---|---|
| 仅SM | 简洁、省空间 | 与其他国家符号冲突风险 | 高频数字界面、用户熟悉符号 |
| 仅TJS | 国际无歧义 | 稍长、占用空间 | 报表、对账、跨境业务 |
| 自动(默认TJS,可切换SM) | 平衡一致性与可读性 | 需设计切换与记忆逻辑 | 通用场景、移动端主推 |

### 5.1 区域设置与语言检测
遵循“系统区域优先,用户显式选择优先”的原则:应用启动时使用系统locale,允许用户在设置中显式选择语言与区域。避免通过IP推断区域,以免造成与用户语言偏好不一致的格式化[^6][^9]。

### 5.2 格式化API与库选择
移动端优先使用原生API:Android的NumberFormatter与iOS的NSNumberFormatter,指定货币代码为TJS并配合不间断空格与分组模式。在需要更细粒度控制时,参考ICU/CLDR的locale模式与消息格式,确保跨端一致[^9][^11]。

### 5.3 币种标识与可读性权衡
货币符号“SM”在不同国家可能存在视觉混淆,推荐在默认态使用ISO代码“TJS”确保无歧义;在空间受限界面可提供“切换到SM”的轻量交互,并记忆用户偏好,保证整应用一致性[^7][^3]。

表8:标识策略决策(示例)
| 场景 | 优先级 | 建议显示 |
|---|---|---|
| 对账单/发票 | 无歧义优先 | 金额 + 空格 + TJS |
| 收银/交易确认 | 速度与简洁 | 金额 + 空格 + TJS,提供SM切换 |
| 营销价签 | 空间紧张 | 可用SM,条件满足时加轻量提示(货币代码) |

### 5.4 边界与异常场景
- 负数:统一使用“-”(连字符)置于金额之前,再空一格加TJS/SM,保持与数字格式的一致性[^5]。
- 零值与空值:零值显示“0,00 TJS”,空值显示“—”(或本地化“空”文案),避免“0,00”误解为未初始化。
- 多货币并列:确保小数位与分组随各自locale;上下文字号与对齐保持一致,防止视觉混淆[^11]。

表9:边界场景清单与通过标准(节选)
| 场景 | 通过标准 |
|---|---|
| 负数 | 符号统一为“-”,位置与空格一致,读数无歧义 |
| 零值 | 显示“0,00 TJS”,不与空值混淆 |
| 空值 | 显示“—”,不抛异常或不显示原始错误码 |
| 多货币并列 | 各币种按各自locale格式化,小数位与分组正确,无串扰 |

## 6. 工程落地清单(可执行)
要将上述规范转化为工程事实,需要围绕格式化函数、模板、文案、测试与配置建立一套可执行清单。

- 统一格式化函数:以“货币代码+locale”为键构建格式化函数;默认使用“金额 + 不间断空格 + TJS”;允许用户偏好切换至“SM”;确保负数、零值、空值统一处理[^9][^11][^5]。
- 模板与文案:在所有价签、报表、交易确认、对账单中统一采用上述格式;对紧凑视图提供降级展示策略(如改用SM并缩短数字分组),但必须保留无歧义的货币标识[^6]。
- 测试矩阵:覆盖Android/iOS、真机/模拟器、tg_TJ/其他locale、在线/离线、边界值与多货币并列;验证不间断空格与逗号是否正确渲染与对齐[^11][^5]。
- 配置文件:维护locale→货币显示映射;明确TJS默认两位小数;在设计规范中记录“4位整数可省略千分位”的可选项[^5]。

## 7. 合规与一致性检查
一致性要求贯穿产品、运营与发布链路:同一模块内金额格式必须一致,跨模块(报表/价签/对账单)使用统一策略;在多区域同时运营时,需在测试报告中显式列出locale覆盖与验证通过项;在需要更高国际化一致性的场景,遵循微软全球化对货币格式化的指导,并记录对ICU/CLDR的版本依赖与兼容性说明[^11][^9]。
